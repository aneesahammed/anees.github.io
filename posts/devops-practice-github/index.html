When you speak about DevOps at scale, being SOC (Service Organization Control) 2 compliant and dealing with hundreds or more projects and engineers, it’s nearly impossible to impart so called “DevOps culture” without putting together team-wide standard practices & policies.

So below are few DevOps practices we follow when we select a candidate application or service for our DevOps onboarding.

Git the single source of truth

Git is a distributed version-control system for tracking changes in source code during software development. So let me start with the single source of truth Git. Adopting a consistent naming convention & repository structure across organization is essential.

Keep all your repository name in small letters

Have - hiphen as word separator: application-service. This is personal preference, and you may separate your words with any other character you want except a space or tab.

Create a develop branch alongside the default master branch.

So that developers can continue checkin their source code to the develop branch. And we can enable develop branch to be automatically deployed on every checkin from the CI orchestrator: Jenkins, CircleCi.

And follow below branching naming convention for other branches.

  • for all the feature branches, follow prefix feature/..
  • for all the bugfix branches follow bugfix/bug-#
  • for all hotfix branches follow hotfix/defect-#

Make sure to üöß protect master & develop branch.

Idea is that, master branch should be deployable and it should reflects the last production released version of the source code.

Ensure every commit message üîó link to Rally/Jira ticket.

This helps in achieving traceability across development lifecycle. And this is the most obvious way to remain compliant for those companies opearting in financial, healthcare & government sectors. And like DevOps Handbook put out “We can generate evidence on demand to demonstrate that our controls are operating effectively, whether to auditors, assessors, or anyone elese working in our value stream”.

This can be enforced at developer end using git hooks that gets triggered whenever there is a commit.

# Commit Message Format
US12345 fixes the nasty divide by zero exception.

Another excellent tool, I recommend to write informative commit messages & streamline the commit process is Commitizen With this you do not need to wait until for a git commit hook to run.

Have a üôà gitignore file & never commit your üì¶ dependencies into source control

What is heaviest thing in universe?. It is nothing but node_modules Keep all the unwanted dependencies, personal configs & IDE generated files in gitignore. Checkout here to create a gitignore file for your stack.

Avoid commiting üîê secrets to source control

Inadvertently checking credentials and other secrets into source control is a major issue. git-secrets is a simple tool that can prevent you from commiting sensitive iformation into git repositories.

As “The 12 factor app” states, a litmus test for whether an app has configs & secrets correctly factored out of the code is whether the codebase could be made open source at any moment, without compromising any credentials.

git-secrets can also scan all historical revisions, so that you can ensure you’ve never checked in any credential.
